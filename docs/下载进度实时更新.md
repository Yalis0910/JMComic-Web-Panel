# 下载进度实时更新 —— 改造开发文档

## 1. 问题分析

### 1.1 现状

当前下载进度只有 0%（任务创建）和 100%（下载完成）两个状态，中间没有任何更新。

原因：
- `download.py:37-38` 的 `callback` 仅在 `download_album` 完成后触发一次 `complete_task`。
- `progress.py:53` 的 `update_progress` 方法已存在但从未被调用。
- 前端轮询 `GET /download/tasks`（`app.js:175-177`）每 2 秒刷新一次，但后端数据从未变化。

### 1.2 核心挑战：获取总页数

要实现进度条，必须先知道总页数。目前：

| 数据源 | 有 page_count? | 说明 |
|--------|---------------|------|
| `JmHtmlClient.get_album_detail()` | **Yes** | 从 HTML 正则解析 `pattern_html_album_page_count` |
| `JmApiClient.get_album_detail()` | **No**（始终为 `'0'`） | API JSON 不含此字段 |
| Web API `GET /api/v1/album/{id}` | 取决于用户配置的 client | 若用户配了 `api` client 则拿不到 |

**结论**：不能依赖用户的 client 配置。需要在下载前临时用 `JmHtmlClient` 单独获取 `page_count`。

## 2. 架构设计

```
用户点击下载
    │
    ├─ POST /download/album → 创建任务、启动线程
    │
    └─ _do_download (后台线程)
         │
         ├─ 1. _build_option() 构建配置
         │
         ├─ 2. 临时创建 JmHtmlClient 获取 page_count
         │        → manager.update_progress(task_id, 0, total_pages)
         │
         ├─ 3. 注入 ProgressPlugin 到 option
         │        → after_photo: 累加 len(photo) → update_progress
         │
         ├─ 4. download_album(...) 执行下载
         │        → 每完成一个章节，插件自动更新进度
         │
         └─ 5. callback → complete_task (已有逻辑)
```

## 3. 详细设计

### 3.1 `jmweb/api/download.py` 改造

#### 改动点：`_do_download`

```python
def _do_download(task_id: str, album_id: str, option_path: str = None, download_type: str = "folder"):
    try:
        option = _build_option(option_path, download_type)
        client = option.new_jm_client(impl='html')
        album = client.get_album_detail(album_id)
        total_pages = int(album.page_count)

        manager.update_progress(task_id, 0, total_pages)
        option._progress_task_id = task_id
        option._progress_total = total_pages
        option._progress_completed = 0

        # 注册进度插件
        JmModuleConfig.REGISTRY_PLUGIN['_progress'] = _ProgressPlugin
        option.plugins.src_dict.setdefault('after_photo', []).insert(0, {
            "plugin": "_progress",
            "kwargs": {},
        })

        def callback(album, dler):
            manager.complete_task(task_id)

        download_album(album_id, option, callback=callback, check_exception=False)

    except Exception as e:
        manager.complete_task(task_id, error=str(e))
```

#### 新增：`_ProgressPlugin`

```python
class _ProgressPlugin(JmOptionPlugin):
    plugin_key = '_progress'

    def invoke(self, photo=None, downloader=None, **kwargs):
        if photo is None:
            return
        task_id = getattr(self.option, '_progress_task_id', None)
        total = getattr(self.option, '_progress_total', 0)
        if not task_id or total <= 0:
            return
        self.option._progress_completed += len(photo)
        manager.update_progress(task_id, self.option._progress_completed, total)
```

**设计说明**：
- 利用 `option` 对象存储进度状态（`_progress_task_id`、`_progress_total`、`_progress_completed`）
- `after_photo` 每完成一个章节触发一次，累加该章节的图片数
- 不对 `after_image` 做 hook，减少调用频率（一章一张 vs 一章几十张）
- `plugin_key` 使用 `_progress` 下划线前缀，表示内部插件不对外暴露
- 不使用 `JmModuleConfig.REGISTRY_PLUGIN` 的全局注册，避免污染（改用临时注册）

## 5. 下载历史持久化

### 5.1 问题

`DownloadManager` 所有任务存储在内存 `_tasks: Dict[str, DownloadTask]` 中，重启服务后全部丢失。用户无法查看上一次服务运行期间的下载记录。

### 5.2 方案：JSON 文件持久化

选用 JSON 方案的原因：
- Python 内置 `json` 模块，**零新增依赖**
- 项目已有 `config/` 目录作为持久化存储目录（`config/option.yml`）
- 数据结构简单（扁平字段），JSON 足够胜任
- 利用现有 `_task_lock` 同步文件读写，无需额外锁机制

#### 存储位置

```
config/download_history.json   ← 与 option.yml 同级
```

#### 数据格式

```json
{
  "tasks": [
    {
      "task_id": "a1b2c3d4e5f6",
      "album_id": "123456",
      "title": "",
      "status": "completed",
      "progress": 100,
      "total_count": 48,
      "completed_count": 48,
      "error": null,
      "created_at": 1234567890.123,
      "download_type": "folder"
    }
  ]
}
```

`thread` 字段不可序列化，写入时排除；读取时 `DownloadTask` 自动取默认值 `thread=None`。

### 5.3 实现细节：`jmweb/utils/progress.py`

#### 新增 `_history_path` 属性与文件路径

```python
import json
from pathlib import Path

_HISTORY_FILE = Path(__file__).parent.parent.parent / "config" / "download_history.json"
_MAX_HISTORY = 200  # 最多保留 200 条记录，防止文件无限膨胀
```

#### 新增 `_save_to_file` 方法

每次任务状态变更后调用，将当前所有任务序列化写入 JSON：

```python
def _save_to_file(self):
    tasks_data = []
    for t in self._tasks.values():
        d = {k: v for k, v in t.__dict__.items() if k != 'thread'}
        tasks_data.append(d)
    # 只保留最近 _MAX_HISTORY 条（按 created_at 降序）
    tasks_data.sort(key=lambda x: x['created_at'], reverse=True)
    tasks_data = tasks_data[:_MAX_HISTORY]
    _HISTORY_FILE.parent.mkdir(parents=True, exist_ok=True)
    _HISTORY_FILE.write_text(
        json.dumps({"tasks": tasks_data}, ensure_ascii=False, indent=2),
        encoding='utf-8'
    )
```

#### 新增 `_load_from_file` 方法

在单例初始化时调用，从文件恢复任务记录。重启前状态为 `running` 的任务自动标记为 `failed`：

```python
def _load_from_file(self):
    if not _HISTORY_FILE.exists():
        return
    try:
        data = json.loads(_HISTORY_FILE.read_text(encoding='utf-8'))
        for item in data.get('tasks', []):
            # 重启后 running 任务已失效（线程已销毁）
            if item['status'] == 'running':
                item['status'] = 'failed'
                item['error'] = '服务重启，任务中断'
                item['progress'] = item.get('progress', 0)
            task = DownloadTask(**item)
            self._tasks[task.task_id] = task
    except (json.JSONDecodeError, KeyError, TypeError) as e:
        import logging
        logging.warning(f"下载历史文件解析失败，将重新开始: {e}")
```

#### 修改 `__new__`：初始化时自动加载

```python
def __new__(cls):
    with cls._lock:
        if cls._instance is None:
            cls._instance = super().__new__(cls)
            cls._instance._tasks: Dict[str, DownloadTask] = {}
            cls._instance._task_lock = threading.Lock()
            cls._instance._load_from_file()  # ← 新增
        return cls._instance
```

#### 修改所有状态变更方法：同步写文件

`create_task`、`update_progress`、`complete_task`、`cancel_task` 四个方法的末尾都需要追加 `self._save_to_file()`：

```python
def create_task(self, album_id: str) -> DownloadTask:
    import uuid
    task_id = uuid.uuid4().hex[:12]
    task = DownloadTask(task_id=task_id, album_id=album_id)
    with self._task_lock:
        self._tasks[task_id] = task
    self._save_to_file()  # ← 新增
    return task

def update_progress(self, task_id: str, completed: int, total: int):
    task = self.get_task(task_id)
    if task:
        task.completed_count = completed
        task.total_count = total
        task.progress = int(completed / total * 100) if total > 0 else 0
        self._save_to_file()  # ← 新增

def complete_task(self, task_id: str, error: Optional[str] = None):
    task = self.get_task(task_id)
    if task:
        task.status = "failed" if error else "completed"
        task.error = error
        task.progress = 100 if not error else task.progress
        self._save_to_file()  # ← 新增

def cancel_task(self, task_id: str):
    task = self.get_task(task_id)
    if task:
        task.status = "cancelled"
        self._save_to_file()  # ← 新增
```

#### 完整的 `progress.py` 改造后结构

```
DownloadManager
 ├── __new__         → 初始化 _tasks、_task_lock，调用 _load_from_file
 ├── create_task     → 创建任务 + _save_to_file
 ├── get_task        → 只读，不变
 ├── get_all_tasks   → 只读，不变
 ├── update_progress → 更新进度 + _save_to_file
 ├── complete_task   → 完成任务 + _save_to_file
 ├── cancel_task     → 取消任务 + _save_to_file
 ├── _save_to_file   → 序列化全部任务写入 JSON
 └── _load_from_file → 从 JSON 恢复，running → failed
```

### 5.4 设计说明

| 关注点 | 做法 |
|--------|------|
| 并发安全 | 写文件在 `_task_lock` 保护的方法内进行；JSON 文件写入是原子操作（`write_text` 整体写入） |
| 文件大小控制 | 最多保留 200 条历史记录，超出时丢弃最早的 |
| 重启恢复 | `running` → `failed`，因为线程已销毁 |
| 序列化安全 | 排除 `thread` 字段（不可 JSON 序列化）；兼容未来新增字段（读取用 `**item` 解包） |
| 性能 | 每次状态变更都写文件。以几秒一次的频率，JSON 文件仅几 KB，开销可忽略 |

## 6. 边界情况处理

| 场景 | 行为 |
|------|------|
| `JmApiClient` 用户 | 预取时临时使用 `JmHtmlClient`，不受影响 |
| 网络错误获取 page_count | 异常被 `try/except` 捕获 → `complete_task(task_id, error)` 标记失败 |
| 下载中途取消 | 插件不再被调用；`manager.cancel_task` 已处理状态 |
| 全本已缓存（无需下载） | `after_photo` 依然会触发 → 进度从 0% → 100%（跳过下载阶段） |
| ZIP 模式 | 同样生效，进度在图片下载阶段更新 |
| 下载失败（部分图片） | 插件只统计成功的章节；最终 `callback` 标记为 `completed`，但 `all_success` 为 False |
| **服务重启** | `running` 任务 → `failed`（"服务重启，任务中断"）；历史记录完整保留 |
| **JSON 文件损坏** | 捕获异常，打印警告，以空记录启动 |
| **历史记录超过 200 条** | 自动裁剪，保留最近的 200 条 |
| **高频率轮询** | 前端 2 秒轮询，后端读取不涉及文件 I/O（从内存返回），写文件仅在状态变更时 |
